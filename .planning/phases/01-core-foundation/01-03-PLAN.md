  <name>Task 1: Implementar Match e Else (pattern matching) para todos os Result types</name>
  <files>Responses/Result.cs</files>
  <action>
    Implementar Match e Else como métodos de pattern matching.

    Para Result&lt;T&gt;:

    ```csharp
    // Match: executa uma das duas funções baseado no estado
    public readonly TResult Match<TResult>(Func<T, TResult> onSuccess, Func<Error, TResult> onFailure)
    {
        return IsSuccess ? onSuccess(_value) : onFailure(_error);
    }

    // Match void (side-effect branches)
    public readonly void Match(Action<T> onSuccess, Action<Error> onFailure)
    {
        if (IsSuccess) onSuccess(_value);
        else onFailure(_error);
    }

    // Else: fallback value quando failed
    public readonly T Else(T fallbackValue)
    {
        return IsSuccess ? _value : fallbackValue;
    }

    // Else com func (fallback computado)
    public readonly T Else(Func<Error, T> fallbackFunc)
    {
        return IsSuccess ? _value : fallbackFunc(_error);
    }
    ```

    Para Result&lt;TValue, TError&gt;:
    - Mesma lógica, mas usando TError no lugar de Error nos delegates de failure

    Para Result (void):
    - Match: `public readonly void Match(Action onSuccess, Action<Error> onFailure)` — sem valor para onSuccess
    - Else não faz sentido para void (não há fallback value) — NÃO implementar

    Match é essencialmente um "switch" funcional:
    - Em success: executa onSuccess com o valor
    - Em failure: executa onFailure com o erro
    - Retorna o valor da função executada

    Else é um "coalesce" funcional:
    - Em success: retorna o valor
    - Em failure: retorna o fallbackValue ou resultado de fallbackFunc
    - Útil para "outra opção" sem lançar exceção

    Cuidados:
    - Manter consistência de delegates entre sync e async (próxima task)
    - Validar que onSuccess e onFailure não são null (lançar ArgumentNullException)
  </action>
  <verify>
    result.Match(x => "ok: " + x, e => "err: " + e.Code) retorna string correta
    result.Else(0) retorna 0 quando failed, valor quando success
  </verify>
  <done>
    Match implementado (com overload void e com retorno) para todos os 3 types
    Else implementado (com value e func) para Result&lt;T&gt; e Result&lt;TValue,TError&gt;
    Null checks nos delegates
  </done>

<task type="auto">
  <name>Task 2: Implementar MapAsync, BindAsync, TapAsync, EnsureAsync e MatchAsync/ElseAsync</name>
  <files>Responses/Result.cs</files>
  <action>
    Implementar variantes assíncronas para todos os métodos de composição.

    Para Result&lt;T&gt;:

    ```csharp
    // MapAsync: transforma valor com operação async
    public readonly async Task<Result<TOut>> MapAsync<TOut>(Func<T, Task<TOut>> func)
    {
        return IsSuccess ? Result.Ok(await func(_value)) : Result.Fail<TOut>(_error);
    }

    // BindAsync: encadeia operação async que pode falhar
    public readonly async Task<Result<TOut>> BindAsync<TOut>(Func<T, Task<Result<TOut>>> func)
    {
        return IsSuccess ? await func(_value) : Result.Fail<TOut>(_error);
    }

    // TapAsync: side-effect async
    public readonly async Task<Result<T>> TapAsync(Func<T, Task> action)
    {
        if (IsSuccess) await action(_value);
        return this;
    }

    // EnsureAsync: guard condicional async
    public readonly async Task<Result<T>> EnsureAsync(Predicate<T> predicate, Error error)
    {
        if (!IsSuccess) return this;
        if (!predicate(_value)) return Result.Fail<T>(error);
        return this;
    }

    // MatchAsync
    public readonly async Task<TResult> MatchAsync<TResult>(Func<T, Task<TResult>> onSuccess, Func<Error, Task<TResult>> onFailure)
    {
        return IsSuccess ? await onSuccess(_value) : await onFailure(_error);
    }

    // ElseAsync
    public readonly async Task<T> ElseAsync(Func<Error, Task<T>> fallbackFunc)
    {
        return IsSuccess ? _value : await fallbackFunc(_error);
    }
    ```

    Para Result&lt;TValue, TError&gt;:
    - Mesma lógica, preservando TError nos delegates de failure

    Para Result (void):
    - TapAsync: `public readonly async Task<Result> TapAsync(Func<Task> action)`
    - MatchAsync: `public readonly async Task MatchAsync(Func<Task> onSuccess, Func<Error, Task> onFailure)` — void return
    - BindAsync: `public readonly async Task<Result> BindAsync(Func<Task<Result>> func)` — encadeia operations void async

    Notas importantes:
    - Estes métodos retornam Task&lt;Result&lt;T&gt;&gt; porque são async
    - O await ocorre DENTRO do método, não fora
    - Short-circuit ainda funciona: failure não executa a func async
    - Usar `ConfigureAwait(false)` se apropriado (mas em biblioteca de resultado, pode ser melhor deixar o chamador decidir)
    - Manter consistência: se existe Map sync, existe MapAsync; se existe Bind sync, existe BindAsync
  </action>
  <verify>
    await result.MapAsync(x => Task.FromResult(x * 2)) funciona em success
    await result.BindAsync(x => SomeFallibleAsyncOp(x)) short-circuits em failure
  </verify>
  <done>
    MapAsync, BindAsync, TapAsync, EnsureAsync implementados para Result&lt;T&gt; e Result&lt;TValue,TError&gt;
    MatchAsync e ElseAsync implementados
    Async variants para Result (void) onde faz sentido
    Short-circuit funciona em failure paths (func async NÃO é chamada)
  </done>

<task type="auto">
  <name>Task 3: Implementar SelectMany para LINQ query syntax</name>
  <files>Responses/Result.cs</files>
  <action>
    Implementar SelectMany para habilitar LINQ query syntax com Result.

    O padrão LINQ requer 2 overloads de SelectMany:

    ```csharp
    // SelectMany para Result<T>
    public readonly Result<TOut> SelectMany<TOut>(Func<T, Result<TOut>> selector)
    {
        return Bind(selector); // SelectMany é alias de Bind
    }

    // SelectMany com projeção final (para query syntax completo)
    public readonly Result<TResult> SelectMany<TOut, TResult>(
        Func<T, Result<TOut>> collectionSelector,
        Func<T, TOut, TResult> resultSelector)
    {
        if (!IsSuccess) return Result.Fail<TResult>(_error);

        var intermediate = collectionSelector(_value);
        if (!intermediate.IsSuccess) return Result.Fail<TResult>(intermediate.Error);

        return Result.Ok(resultSelector(_value, intermediate.Value));
    }
    ```

    Para Result&lt;TValue, TError&gt;:
    - Mesma lógica, preservando TError

    Isso habilita:
    ```csharp
    var result = from x in Result.Ok(5)
                 from y in Result.Ok(10)
                 select x + y;
    // result == Result.Ok(15)
    ```

    O compilador C# traduz query syntax para chamadas SelectMany.

    IMPORTANTE:
    - SelectMany é essencialmente Bind (railway-oriented)
    - O segundo overload com resultSelector permite combinar múltiplos Results
    - Se qualquer operação intermediária falhar, o resultado final é failure
    - Manter consistência com Bind existente
  </action>
  <verify>
    Query syntax compila: `from x in Result.Ok(5) from y in Result.Ok(10) select x + y`
    Retorna Result.Ok(15) quando todos são success
    Retorna Result.Fail quando qualquer operação intermediária falha
  </verify>
  <done>
    SelectMany implementado com ambos os overloads
    LINQ query syntax compila e funciona
    Short-circuit em falhas intermediárias
  </done>
